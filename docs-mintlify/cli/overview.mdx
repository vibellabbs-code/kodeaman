The CLI (`apps/cli`) is the primary entry point. It wires every product layer together through `commander` commands.

## Invoking the CLI

During local development, run through pnpm from the repo root:

```bash
pnpm --filter @aspidasec/cli start -- <command> [options]
```

Where a global binary named `aspidasec` is installed, the same commands apply directly:

```bash
aspidasec <command> [options]
```

<Note>
  Docs below show the `aspidasec <command>` form for brevity. Substitute the `pnpm --filter @aspidasec/cli start --` prefix when running from a source checkout.
</Note>

## Shared conventions

| Convention | Detail |
| --- | --- |
| Config file | `.aspidasec.yml` in the working directory, override with `-c, --config`. |
| Language | `-l, --language en\|id`, defaults to the config value. |
| Format | `-f, --format`, accepted values vary per command. |
| Verbose | `-v, --verbose` enables detailed output where supported. |

## Exit codes

| Code | Meaning |
| --- | --- |
| `0` | Scan completed; no findings at or above `failOnSeverity`. |
| `1` | Findings at or above `failOnSeverity`, or a scan error. |

These codes make AspidaSec safe to gate CI on — fail the build only when findings cross your configured threshold.
