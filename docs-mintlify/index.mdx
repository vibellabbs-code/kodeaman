## Why AspidaSec

Most scanners detect issues but fail developers at the last mile: noisy output, duplicated findings, no prioritization, and no safe fix. AspidaSec focuses on the parts that make website security *actionable*.

## The golden path

The first workflow is intentionally narrow — point AspidaSec at a project or a URL and get a prioritized report.

<CodeGroup>
```bash Local project
aspidasec scan .
```

```bash Deployed site
aspidasec scan https://example.com
```

```bash OWASP report
aspidasec owasp-scan --format html --output report.html
```
</CodeGroup>

Under the hood, a scan moves through one clear pipeline:

<Steps>
  <Step title="Collect">
    Run dependency audit, OWASP-oriented static checks, dynamic scan, route crawling, and basic secret/config review.
  </Step>
  <Step title="Normalize">
    Map every scanner result into a single finding schema and deduplicate equivalents.
  </Step>
  <Step title="Prioritize">
    Score by severity, confidence, exploitability, exposure, affected file/endpoint, and fix availability.
  </Step>
  <Step title="Guide">
    Attach practical remediation, safe code examples, and references.
  </Step>
  <Step title="Report">
    Render Markdown, HTML, JSON, SARIF, or a PR/MR comment.
  </Step>
</Steps>
